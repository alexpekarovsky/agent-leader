🏷 Naming Convention: Description Tags (Not Renaming)

Don't rename tools — add [ROLE][SAFETY] tags to descriptions. Same LLM benefit, zero breakage.

Why Description Tags Win
LLMs read descriptions before calling tools. A [WORKER] tag lets worker agents skip manager tools instantly. Zero tool name changes needed.
Why Renaming Loses
Breaks: tools.json contract, 67 README refs, 42 test refs, 4 startup templates, command bundles, guide playbook, all active agent sessions.

Tag Format

[MANAGER][MUTATING] Run one manager cycle: validate reports, reassign stale tasks...
[WORKER][SAFE] Read the generated spec file for implementation guidance.
[OPERATOR][SAFE] Show status dashboard with progress bars and team roster.
[INFRA][DESTRUCTIVE] Stop headless supervisor lanes for this project.
[SHARED][MUTATING] Update last-seen heartbeat for agent.
[INTEGRATION][SAFE] Normalize a GitHub CI payload into orchestration summary.

Safety Levels (3)

SAFERead-only, idempotent. No state changes.
MUTATINGChanges state but reversible. Creates/updates tasks, agents, events.
DESTRUCTIVEIrreversible or process-affecting. Starts/stops processes, deletes artifacts.

Replaces get_roles. guide_payload() depends on role info. Operators expect to query roles independently.

Removed tool names redirect to merged equivalents. get_roles → status(format="roles"), register_agent → heartbeat(auto_register=true), etc.
