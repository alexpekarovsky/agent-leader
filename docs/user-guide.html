How To Run The System From Codex CLI

This is a user guide in plain language. You control everything from this chat. Codex runs the orchestrator steps for you.

Flow A: Interactive (chat-driven, step by step) Flow B: Headless swarm (continuous autonomous cycles) Project example: /Users/alex/my-api

What You Need Before Starting

  • Your project exists locally (example: /Users/alex/my-api).
  • Orchestrator MCP is reachable from this Codex session.
  • You know your goal in one sentence (for example: "add /health endpoint and tests").
If connection fails, ask: check orchestrator status and tell me what is down.

Who Does What

Leader (Codex)

  • Owns task lifecycle.
  • Connects team members.
  • Creates tasks, validates reports, accepts or requests fixes.

Wingman / Worker

  • Implements code changes.
  • Adds tests.
  • Submits delivery report with test results.

Flow A: Interactive (Recommended For Manual Control)

Step 1

Tell Codex to enter leader mode and connect team

Say in chat: "You are the leader. Connect to team and show connection status."

Codex should reply with current roles and active team members.

Step 2

If you also have a wingman session, attach it

In wingman session, say: "You are team member. Connect to leader and report status."

If the same identity is already leader, Codex explains why attach is blocked and what to switch.

Step 3

Give goal in human language

Say: "Use project /Users/alex/my-api. Please add a /health endpoint and tests. Make a minimal plan, then execute."

Codex converts this to tasks and assigns workers.

Step 4

Steer during execution

Say: "Status now." or "Show blockers only." or "Validate reported tasks now."

You can intervene at any time: pause, reprioritize, or request review focus.

Step 5

Close the run

Say: "Close completed tasks and summarize what changed, what was tested, and any risks."
Done means: tasks are done, no open blockers, no open bugs.

Flow B: Headless Swarm (Recommended For Throughput)

Step 1

Start autonomous run in plain language

Say: "Start headless swarm for /Users/alex/my-api to add /health endpoint and tests. Use leader codex and available workers. Send progress every cycle."

You do not need to type low-level parameter blocks unless you want to.

Step 2

Let Codex seed and run tasks

Codex will create tasks, workers claim them, code is implemented, tests run, and reports are submitted.

1) Connection and role checks.
2) Task creation from your mission.
3) Worker claim and execution loops.
4) Leader validation and possible fix-loop.
5) Completion summary.
Step 3

Observe without micromanaging

Say: "Report every cycle. Alert me only on blockers, failures, or regressions."

You can also ask for a short table of in-progress and done tasks at any time.

Step 4

Recover if stalled

Say: "Diagnose swarm health and recover automatically. Show only actions taken and unresolved blockers."
Done means: mission tasks done, tests green, no unresolved blockers.

New: One Leader, Multiple Teams (Headless)

You can now run one leader and route workers into separate teams using team_id.

  • Create tasks with team_id, project_root, and project_name.
  • Start workers with matching team lanes (for example team-api and team-web).
  • Use orchestrator_list_tasks(team_id=\"...\") to monitor each team separately.
Example: "Run one leader. Put Codex+Wingman on team-api in /Users/alex/my-api and Gemini on team-web in /Users/alex/my-web."

Headless Command Pattern

./scripts/autopilot/supervisor.sh start \
  --project-root /Users/alex/claude-multi-ai \
  --leader-agent claude_code \
  --codex-team-id team-api \
  --wingman-team-id team-api \
  --gemini-team-id team-web \
  --codex-project-root /Users/alex/my-api \
  --wingman-project-root /Users/alex/my-api \
  --gemini-project-root /Users/alex/my-web

If you need more workers, add --extra-worker name:cli:agent:team_id:project_root[:lane].

End-To-End Example (Small API Project)

Interactive Example
You: "You are the leader. Connect to team and show status."
You: "Use /Users/alex/my-api. Add /health endpoint and tests. Keep scope small."
You: "Assign coding to wingman. Leader should review and validate."
You: "Status every 2 minutes. Blockers only if urgent."
You: "When done, commit and summarize files changed + tests run."
Headless Example
You: "Start headless swarm for /Users/alex/my-api. Mission is /health endpoint plus tests."
You: "Keep cycles running automatically and report each cycle briefly."
You: "If something fails, retry once and then ask me only with options."
You: "When complete, produce final review summary and close tasks."

If You Have Only Leader + Wingman

This setup works well:

  • Wingman: implementation + unit tests + delivery report.
  • Leader: code review + regression checks + accept/reject + task state management.
Use this sentence: "Wingman writes code and tests. Leader only reviews, validates, and controls task lifecycle."

Common Problems And Exact Fix Prompts

Transport closed

"Check orchestrator connectivity and tell me what service is down. Retry connect once service is up."

No claimable task

"Create new tasks for current mission and assign owners, then show queue."

Role mismatch

"Show current leader and team members. If needed, switch leader so wingman can attach."

Too many noisy updates

"Switch to concise updates: blockers, failures, and completion only."

What To Say First (Cheat Sheet)

  • Interactive: "You are the leader. Connect to team. I will steer step by step."
  • Headless: "Start headless swarm for this project and keep it running until mission is done."
  • Status: "Give me current status: in progress, done, blockers, and next action."
